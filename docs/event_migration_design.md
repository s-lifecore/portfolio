# イベント管理のMarkdown（MD）移行設計書

現在のFirebase（Firestore）によるイベント管理を、ローカルでも可視化・編集しやすく、かつ表示速度が極めて高速な**Markdown（MD）ファイルベースの管理**に移行するための設計書です。

---

## 1. 移行の背景と目的

- **脱Firebaseによるコスト・管理の最適化**: イベント管理のためだけにFirebaseを維持するコストと手間を削減します。
- **高速なページ反映とロード**: ビルド時にMarkdownファイルをパースして静的生成（SSG）するため、`/joined` ページの表示速度が劇的に向上します。
- **ローカルでの可視化・編集性の向上**: ローカルの使い慣れたMarkdownエディタ（VS CodeやObsidianなど）で、イベント一覧を直接閲覧・編集・検索できるようになります。
- **管理者画面（`/admin/events`）の維持**: これまで通り、ブラウザ上の管理者画面から直感的にイベントの追加・編集・削除が行えるようにします。

---

## 2. システムアーキテクチャ（ハイブリッド方式）

本番環境（Vercel等）とローカル開発環境で、書き込み処理を自動的に切り替える**ハイブリッド書き込み方式**を採用します。

```
【ローカル開発環境 (development)】
管理者画面 ──> /api/events ──> 直接ローカルの fs (content/events/*.md) を書き換え (即時反映)

【本番環境 (production)】
管理者画面 ──> /api/events ──> GitHub API ──> リポジトリへ自動コミット ──> Vercel自動ビルド ──> ページ反映
                                                        └─> ローカルで `git pull` して同期
```

### 環境ごとの動作仕様

| 項目 | ローカル開発環境 (`development`) | 本番環境 (`production`) |
| :--- | :--- | :--- |
| **データの読み込み** | ローカルの `content/events/` 内の `.md` ファイルを `fs` で読み込み | ビルド時に同梱された `.md` ファイルを `fs` で読み込み（高速SSG/ISR） |
| **データの書き込み・削除** | **ローカルファイルシステム (`fs`) に直接書き込み**。エディタにも即時反映されます。 | **GitHub API経由でリポジトリに直接コミット**。履歴がGitログに残り、自動デプロイが走ります。 |
| **認証 / 必要な環境変数** | なし（ローカルの管理者パスワードのみ） | `GITHUB_TOKEN` (リポジトリへの書き込み権限を持つパーソナルアクセストークン) |

---

## 3. Markdownファイル設計

イベントごとに1つのMarkdownファイルを作成します。メタデータは**Frontmatter（YAML形式）**で管理し、本文にイベントの「説明（description）」を記述します。

### ディレクトリ構成
```text
portfolio/
├── content/
│   └── events/
│       ├── 2027-03-19-e1773894107798.md
│       └── [date]-[id].md
```

### ファイル名規則
`content/events/{YYYY-MM-DD}-{id}.md`
- 例: `content/events/2027-03-19-e1773894107798.md`
- 日付とユニークIDを組み合わせることで、ファイル名の重複を防ぎ、時系列でのソートを容易にします。

### Markdownフォーマット（例）
```markdown
---
id: "e1773894107798"
title: "サンプル"
date: "2027-03-19"
tags:
  - "participated"
host: "主催者名"
venue: "会場名"
url: "https://lifecore.jp"
reviewUrl: "https://note.com/link_sudo"
createdAt: "2026-05-29T00:00:00.000Z"
updatedAt: "2026-05-29T00:00:00.000Z"
---

このような感じで、今後追加していきます。
本文に詳細な説明やメモを記述できます。
```

## 4. 移行ステップ

1. **パッケージの導入**:
   - Markdownのフロントマターをパースするため、軽量な `gray-matter` を導入します。
2. **既存データの移行**:
   - 現在Firestoreにある、または `lib/events.json` にある既存のイベントデータをMarkdownファイルに一括変換する移行スクリプトを作成・実行します。
3. **データ読み込み共通ライブラリの作成 (`src/lib/eventsMarkdown.js`)**:
   - Markdownファイルを読み込み、パースしてソート済みのイベント一覧を返す関数を実装します。
4. **APIルート (`src/app/api/events/route.js`) の刷新**:
   - GET: `eventsMarkdown.js` からデータを取得して返します。
   - POST / PUT / DELETE: ローカル環境なら `fs`、本番環境なら `GitHub API` を叩くように書き換えます。
5. **`/joined` ページの書き換え**:
   - Firestore依存を完全に排除し、`src/lib/eventsMarkdown.js` から直接データを読み込むように変更します（サーバーコンポーネントとしてビルド時に高速レンダリング）。
6. **動作検証**:
   - ローカルでの読み書き、削除が正常に動作することを確認します。
